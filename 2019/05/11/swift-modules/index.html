基于Swift的模块化开发实践方案

Posted on 2019-05-11 | In ios

前言：

本文主要讲解了在使用Swift开发过程中，模块化工程架构的使用方式，主要采用Cocoapods的方式的模块化方案，详细讲解iOS开发如何进行模块化划分与相关问题。

一、为什么要做模块化

程序猿都知道最基本的代码设计原则DRY（Don't Repeat Yourself）。每一个工程都会自己的封装代码与架构设计，当你写几天代码后，会把常用的重复代码封装起来单独放到一个文件夹中，实现代码的复用。但是每个人都有自己的封装，如果把这个文件统一放到一处，其他开发人员直接引用，可以更快捷的开发，更好的共享资源，避免重复造轮子。

二、模块设计原则

  1. 模块层级，从上到下依赖原则，决不能出现下层模块依赖上层模块
  2. 每个模块业务独立，功能明确
  3. 底层模块，接口稳定性高，功能高度复用，例如网络请求模块
  4. 提升模块复用度、自完备性

三、模块化开发优缺点

优点：

  1. 提高代码的复用度，还可以实现真正的功能复用，比如同样的功能模块如果实现了自完备性，可以在多个app中复用
  2. 业务隔离，对不同的业务模块可以进行物理隔离（通过 git 仓库权限控制），进一步提升代码的稳定性。
  3. 模块化对代码的封装性、合理性都有一定的要求，提升开发同学的设计能力。
  4. Swift语法版本升级，可以单个模块单独语言升级，无需整个项目语法升级完成后才可以编译

缺点：

  1. 入门门槛较高，新手入门需要的成本也更高
  2. 工具的使用成本，团队间和模块间的配合成本升高

四、实践步骤

4.1 层级划分设计

当我们确定需要对项目进行模块化了，我们第一个要解决的问题就是如何拆分模块。这是一个见仁见智的问题，没有太明确的划分边界，大致做到每个模块只包含一个功能即可，具体实施还是要根据实际情况权衡。

当我们写一个类的时候，我们会谨记高内聚，低耦合的原则去设计这个类，当涉及多个类之间交互的时候，我们也会运用SOLID原则，或者已有的设计模式去优化设计，但在实现完整的业务模块的时候，我们很容易忘记对这个模块去做设计上的思考，粒度越大，越难做出精细稳定的设计，我暂且把这个粒度认为是组件的粒度。

模块可以是个广义上的概念，并不一定是页面跳转，还可以是其他不具备UI属性的服务提供者，比如日志服务，内存管理服务等等。说白了我们目标是站在更高的维度去封装功能单元，把多个功能单元组合在一起形成一个更大的功能单元，也就是组件。对这些功能单元进行进一步的分类，才能在具体的业务场景下做更合理的设计。

4.2 业务模块解耦方案选型

iOS业内业务模块解耦方式主要有三个方式

  1. URL Router
  2. Target Action
  3. Protocol Class

4.2.1 URLRouter

URLRouter可以参考开源界比较成熟的方案MGJRouter、JLRouter、HHRouter，把每一个模块以一个URL的形式作为独立存在的界限范围。URLRouter需要在内存中维护一张URL表作为key来实现对应的push或者唤起对应的模块。类似get方式的openURl它在传递参数过程中不能传递类似于对象这样的数据，只能传递普通类型数据。

4.2.2 Target Action

TargetAction也是基于苹果原生的Runtime的优势，其优势很大在于可以做到无需加载内存，并且在调用中没有相关的依赖。在调用中的缺陷在于只有在运行池中才可以知道对应的组件是否存在，若不存在会造成对应的Crash。另外代码中会出现HardCode,代码变得晦涩难懂，调用其他模块方法没有提示。使用CTMediator 分类的方式解决调用提示问题也有点重复造轮子的意思。

4.2.3 Protocol class

Protocolclass是利用原始class以及对应的Protocol用到的Serviceinterface去进行实现以及接口的分离，让大部分的业务方只需要基于Protocol的interface进行调用。缺点是其过程中会涉及到内存占用的问题，另外调用方和响应方都必须依赖于同一个service协议文件，每个service协议需单独一个库，pod库数量大量增加。

4.2.4 方案选择

每个方案都有各自优缺点，都能解决模块化的解耦问题。但是现在针对swift的模块化开发，建议采用面向协议调用的URLRouter方式。
通过定义一套协议来实现模块间的通信，协议现成的，那就是URL跳转协议，基本满足需要，简单易上手，基本上现在很多的App架构里面都会有“统一跳转” 这一套东西的，这个不光是对模块解耦有帮助，对于统一化运营都是有极好的帮助的，比如app里面的任何页面，或者任何操作都是通过一个URL来唤起的话，这样是不是就把各个复杂的业务之间解耦了呢，通信都使用URL.

4.3 模块接口设计（待完善）

4.4 模块代码设计

业务模块采用MVVM，其他组件可以采用MVC、MVP等，基础组件的接口建议采用苹果官方的接口设计方案。

4.4.1 架构模式 - MVVM

  1. 低耦合。视图（View）可以独立于Model变化和修改，一个ViewModel可以绑定到不同的”View”上，当View变化的时候Model可以不变，当Model变化的时候View也可以不变。
  2. 可重用性。把一些视图逻辑放在一个ViewModel里面，让很多view重用这段视图逻辑。
  3. 独立开发。开发人员可以专注于业务逻辑和数据的开发（ViewModel）。
  4. 可测试。测试可以针对ViewModel来写。

4.4.3 View 与 Controller - 五层结构规范

制定View与Controller规范的重要性在于：

  1. 提高业务View与Controller层的可读性和可维护性
  2. 防止业务代码对架构产生腐蚀
  3. 保持架构可持续发展，不受外界干扰
  4. 结构的稳定性，确保传承

五层结构如下

  1. lift cycle,方法按照生命周期依次排序
  2. private methods 系统私有方法、封装方法
  3. event response 事件交互方法、网络数据获取方法
  4. interface 对外与对内的接口,内部接口必须使用private修饰
  5. getter 界面元素的初始化方法，必须私有不对外开放，统一添加修饰 private lazy var

五、常见问题

5.1 如何合理的模块业务分离

首先保证业务模块的独立性，如果使用其他模块业务，最好使用协议路由调用。如果调用的功能没有做成模块，可以在现有的主工程中将使用功能进行路由注册。

5.2 模块如何资源导入

  1. 针对底层组件与基础组件，建议使用bundle方式引入，可以参照MJRefresh中bundle的使用方式。
  2. 业务模块资源较多，建议使用Assets方式引入，可以参照PKHUD的资源使用

5.3 使用到objective-c语言的功能代码，如何处理

  1. 如果是功能代码相对独立，建议单独封装成一个组件
  2. 如果功能代码可以被Swift所取代，建议改写为Swift功能代码

5.4 第三方的静态库，如果下潜

  1. 如果是Swift的静态库，可以在podspec添加vendored_libraries指定静态库
  2. 如果是objective-c的第三方静态库，首先制作成objective-c的组件，然后去引用
